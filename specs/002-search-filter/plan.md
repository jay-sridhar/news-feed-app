# Implementation Plan: In-Feed Keyword Search

**Branch**: `002-search-filter` | **Date**: 2026-03-01 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/002-search-filter/spec.md`

## Summary

Add a client-side search bar that filters the currently active category feed in real time
as the user types. The bar appears below the tab bar whenever articles are loaded
(`status === 'success'`), is hidden during loading and error states, resets on every tab
switch, and shows a "no results" message when the query matches nothing.

**Technical approach**:
- Expose `allArticles` from `useFeed` (additive change — one new return field)
- Add `SearchBar` controlled component (new file)
- Hold `query: string` state in `FeedContainer` with a `useEffect` reset on `activeCategory` change
- Filter `allArticles` when query is non-empty; suspend infinite scroll while filtering
- No new dependencies, no new context providers, no backend calls

## Key Design Decisions (from research.md)

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Filter scope | `allArticles` (full set, not paginated slice) | Matches fetched-but-hidden articles; better UX |
| State location | `useState` in `FeedContainer` | Minimum scope; no new context needed |
| Debounce | None | In-memory filter at RSS scale is < 0.5 ms; debounce adds perceptible lag |
| Match fields | `title` + `sourceName` | Both present on every article; body text not available |
| SearchBar visibility | `status === 'success'` only | Hidden during loading/error (FR-010) |
| New dependencies | Zero | Native `String.prototype.includes` is sufficient |

## Implementation Notes

### `useFeed.ts` change (minimal)

Add `allArticles` to the return:

```typescript
// Before (line 103):
return { articles: visibleArticles, status, error, lastRefreshed, hasMore, loadMore, retry }

// After:
return { articles: visibleArticles, allArticles, status, error, lastRefreshed, hasMore, loadMore, retry }
```

Also add `allArticles: NewsArticle[]` to the `UseFeedReturn` interface. No other logic changes.

### `SearchBar.tsx` structure

Fully controlled component — no internal state:

```
<div>                           ← full-width container with relative positioning
  <input type="search" ...>    ← text input, value + onChange wired to props
  {value && <button onClick={onClear}>×</button>}   ← clear button, trailing edge
</div>
```

Tailwind classes to respect:
- `w-full` — stretches to container width
- `min-h-[44px]` — accessible touch target
- `px-4 py-2` — comfortable padding
- `pr-10` — right padding to avoid text overlapping clear button
- `rounded-lg`, `border`, `bg-white` — card-style appearance

### `FeedContainer.tsx` changes

### Infinite scroll behaviour during search

When `query` is non-empty:
- `displayedArticles` is derived from `allArticles` — the entire fetched set
- `ScrollSentinel` is **not rendered** (no point paginating a small filtered list)
- `hasMore` from `useFeed` remains unchanged but is unused while filtering

When `query` is empty:
- `displayedArticles` equals the existing `articles` (paginated slice)
- `ScrollSentinel` renders as before — no behaviour change

### "No results" state

When `displayedArticles.length === 0` and `status === 'success'` and `query.trim() !== ''`,
render a simple inline message rather than the existing `<ErrorState>`:

```tsx
<p className="text-center text-gray-400 py-8 text-sm">
  No articles match "{query.trim()}"
</p>
```

This is distinct from `<ErrorState>` (which has a retry button). No retry makes sense
for a search that found nothing.
